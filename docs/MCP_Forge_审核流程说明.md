# MCP Forge 审核流程说明

## 1. 先说结论

在 `MCP Forge` 里，不是简单地“每一个 MCP 都从头到尾人工重审一遍”，而是：

`每一个正式要发布的 MCP 资产，都要经过治理流程；但审核重点分层，不是所有内容都用同样强度审核。`

更准确地说，平台审核的不是一个笼统的 “MCP 名称”，而是分成 3 层：

1. `审核 AI 识别出来的接口候选是否可信`
2. `审核 Tool 设计是否合理`
3. `审核 MCP 资产是否达到可发布状态`

---

## 2. 到底审核什么

### 2.1 第一层：审核 AI 识别结果

这一层审核的是：

- AI 从资料中抽取出来的接口候选是否正确
- 接口分类是否正确
- 字段语义是否正确
- 敏感字段识别是否正确
- 不同来源之间是否存在冲突

这一步审核的对象是：

`CandidateAsset`

也就是 AI 初判后的候选接口资产，不是最终 MCP。

### 2.2 第二层：审核 Tool 设计

这一层审核的是：

- 一个 Tool 是否该存在
- 一个 Tool 是否合并过头
- 一个 Tool 是否拆得过碎
- Tool 输入参数是否合理
- 敏感字段是否应该暴露给 Tool
- Tool 的可见性、权限范围、业务规则是否合理

这一步不是在审“接口有没有识别出来”，而是在审：

`识别出来以后，应该如何被组织成可交付的 Tool`

### 2.3 第三层：审核 MCP 是否可以发布

这一层审核的是：

- MCP 是否已经经过测试
- 审核任务是否已经处理完
- 人工初筛是否通过
- 验收清单是否完成
- 敏感字段处理是否确认
- 回滚方案是否具备

这一层更像：

`发布验收`

不是继续校对 AI 识别本身。

---

## 3. 是不是每一个 MCP 都要审核

答案是：

`每一个正式发布的 MCP 都要经过审核链路，但不是每一步都必须人工深审。`

平台里的审核强度分为 3 级：

### 3.1 自动通过

适用于：

- 高置信度
- 低风险
- 没有敏感字段
- 与已有标准一致

这类候选可以自动进入下一步，不需要重人工校对。

### 3.2 单人审核

适用于：

- 分类模糊
- 字段语义不稳定
- 低置信度
- 业务解释不充分

这类候选需要一个负责人做判断。

### 3.3 双人审核

适用于：

- 涉及金额
- 涉及身份信息
- 涉及账号信息
- 涉及权限边界
- 涉及写操作
- 涉及跨系统关键映射

这类候选不能只靠一个人拍板，必须升级更高强度审核。

---

## 4. 正确理解“审核”的顺序

可以把整条链路理解成：

`AI 识别 -> 人工初筛 -> Tool 审核 -> MCP 封装 -> 发布前验收 -> 正式发布`

不是：

`先生成 MCP -> 再统一看一眼`

也就是说，审核应该尽量前置，不要等到 MCP 已经封装完成才发现基础识别就错了。

---

## 5. 审核的具体流程

## Step 1：资料接入

输入资料包括：

- Swagger / OpenAPI
- SQL DDL
- 字段说明
- 业务文档
- 历史接口资料

目标是把分散资料统一成可被 AI 处理的来源。

## Step 2：AI 初判

AI 负责生成：

- 接口候选
- 业务分类建议
- 字段语义识别
- 敏感字段候选识别
- 相似接口聚类
- 置信度和解释信息

注意：

`AI 只负责建议，不负责最终定稿。`

## Step 3：人工初筛

系统先判断当前候选是否需要人工介入。

通常会触发人工门的情况有：

- 风险等级高
- AI 置信度低
- 命中敏感字段
- 数据库与 OpenAPI 冲突
- 文档与接口语义不一致
- 业务域归属不清

人工初筛的动作通常是：

- approve
- reject
- modify

## Step 4：Tool 审核

通过初筛后，进入 Tool 设计审核。

这一步重点确认：

- Tool 是否应该拆分
- Tool 是否合并过头
- 参数是否应该删减
- 敏感字段是否应该隐藏
- 权限范围是否需要缩小
- 是否需要补充业务规则说明

这一步通过后，候选结果才适合进入 MCP 资产封装。

## Step 5：封装成 MCP

经过前面审核确认后，平台才能把确认过的 Tool 组装成 MCP 资产。

这一步不是“审核完成”，而是：

`进入可测试状态`

## Step 6：发布前验收

在正式写入发布资产前，还要经过 3 道阻断检查：

1. `审核任务清零`
所有 open review tasks 必须处理完成。

2. `人工初筛通过`
如果人工初筛明确 reject，则不能继续发布。

3. `验收清单通过`
所有发布前验收项必须完成。

当前验收重点通常包括：

- 业务结果正确
- 敏感字段已处理
- 权限范围已确认
- 写操作风险已确认
- 交付说明已补齐
- 回滚方案可用

## Step 7：正式发布

只有在前面的人工审核和发布验收都通过后，候选结果才会进入：

`PublishedAsset`

这时它才算真正的正式 MCP 资产。

---

## 6. 所以审核重点到底在哪

如果用一句话概括：

`前审重点审“AI 识别 + Tool 设计”，后审重点审“MCP 是否达到可发布标准”。`

也可以拆成两句：

- `前面审对不对`
- `后面审能不能上`

---

## 7. 管理端应该如何表达这套逻辑

如果放在管理端页面里，建议不要写成抽象概念，而要直接告诉管理员：

### 7.1 审核对象说明

- 当前审核的不是最终 MCP 名称，而是 `候选接口` 和 `Tool 设计`
- MCP 封装后还需要经过 `发布前验收`

### 7.2 当前任务说明

- 先确认 AI 识别是否可信
- 再确认 Tool 是否合理
- 最后确认 MCP 是否可发布

### 7.3 阻断说明

以下情况不能发布：

- 审核任务未处理完
- 人工初筛未通过
- 验收清单未完成

---

## 8. 推荐给团队的统一说法

后续无论是在产品文档、后台原型还是汇报里，建议统一用下面这段话：

> MCP Forge 不是对每个 MCP 做笼统人工校对，而是对 `AI 识别候选`、`Tool 设计` 和 `发布前 MCP 验收` 三个层次分别治理。  
> 先确认候选是否可信，再确认 Tool 是否合理，最后确认 MCP 是否达到可发布标准。

---

## 9. 最终结论

最终可以浓缩成一句话：

`先审候选和 Tool，审过后才能封装成 MCP；封装后发布前，还要再验收这个 MCP 是否真正可上线。`
