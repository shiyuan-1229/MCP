# MCP Forge 管理员端：强引导导航重构设计

**日期：** 2026-07-18  
**状态：** 已确认，待进入实施计划  
**适用角色：** 平台管理员（内部交付/运营人员）

## 1. 目标与边界

MCP Forge 的管理员端要支持交付人员把客户资料持续推进为可测试、可发布、可交付的 MCP 资产。当前导航将许多连续的技术步骤并列为一级入口，用户需要自行判断先后顺序，功能感知冗余且页面引导不足。

本次重构目标：

1. 将后台从“模块菜单”改为“任务驱动的资产生产工作台”。
2. 让管理员登录后立即知道最该处理的项目、阻断原因和唯一下一步。
3. 保留现有全部业务功能、数据和权限；仅重构信息架构、入口层级、页面引导与跳转路径。
4. 让线上异常自动回流到正确的生产阶段，避免用户在模块间猜测修复路径。

不在本次范围内：改变 MCP 识别、审核、发布、监控或交付的业务规则本身；删除现有数据；改变客户侧导航。

## 2. 用户与核心任务

核心用户为内部交付/运营人员。他们最高频、最高价值的工作是：将客户提交的业务资料推进为可交付的 MCP 资产。

核心路径：

`资料接入 → AI 识别与审核 → 组装 MCP 资产 → 测试与发布 → 交付确认`

系统需要在每一阶段回答三个问题：

1. 当前处于什么状态？
2. 还缺什么才能继续？
3. 点击哪个操作可以推进？

## 3. 导航信息架构

一级导航分为“今日工作、资产生产、辅助工作”三组，共九个入口。

| 分组 | 一级入口 | 职责 |
| --- | --- | --- |
| 今日工作 | 今日待办 | 按交付影响和紧急程度排序展示待处理项目、阻断项与继续处理入口。 |
| 资产生产 | 资料接入 | 上传、登记和检查业务资料、数据源、知识资料。 |
| 资产生产 | AI 识别与审核 | 查看 AI 识别结果、候选能力与人工初审；以页内阶段管理。 |
| 资产生产 | 组装 MCP 资产 | 确认 Tool 边界、生成 Tool 草稿、确认 MCP 组成并生成 MCP 草稿；以页内阶段管理。 |
| 资产生产 | 测试与发布 | 沙箱调试、质量验证、版本管理、灰度发布与回滚。 |
| 资产生产 | 交付确认 | 打包交付物、验收材料、下载记录与交付确认。 |
| 辅助工作 | 项目资产库 | 跨项目查看项目、资产、版本与当前阶段，支持回到相应任务。 |
| 辅助工作 | 治理运营 | 调用监控、Trace、异常、策略、审计和用量；问题转化为生产任务。 |
| 辅助工作 | 平台设置 | 凭证、权限、知识资料配置、计费及 AI/WorkBuddy 配置。 |

## 4. 统一页面骨架

所有资产生产页面使用以下结构，保证用户无需重新学习每个页面：

1. **项目/资产上下文：** 固定显示项目、MCP 资产和当前阶段。
2. **阶段进度轨道：** 显示五个阶段；已完成可回看，当前阶段高亮，后续阶段可查看但受准入条件约束。
3. **行动摘要：** 显示当前状态、阻断项、缺失条件及其对交付的影响。
4. **当前工作区：** 保留原始列表、表单、审核、编辑和详情能力。
5. **唯一主按钮：** 明确表达当前可推进动作，例如“提交 AI 识别”“生成 Tool 草稿”“执行沙箱测试”。次要操作收入更多操作区。

典型提示：

> 当前 MCP 已具备 3/4 个必需 Tool，缺少“订单详情查询”的参数边界确认。完成后可直接生成“销售查询 MCP v0.1”。

## 5. 各页引导行为

| 页面 | 页首引导 | 主按钮示例 |
| --- | --- | --- |
| 今日待办 | 优先处理哪个项目、影响什么、缺几步。 | 继续处理 |
| 资料接入 | 资料完整度、连接健康、缺失资料。 | 提交 AI 识别 |
| AI 识别与审核 | 识别结果、待审核数、高风险原因、审核后产物。 | 批量确认 / 去处理高风险 |
| 组装 MCP 资产 | 已确认 Tool、缺失边界、组成完整度。 | 确认 Tool 边界 / 组装 MCP |
| 测试与发布 | 发布准入、失败原因、版本状态。 | 执行测试 / 修复后重试 / 发布版本 |
| 交付确认 | 交付物完整度、验收证据、客户可见性。 | 补齐交付包 / 发起交付确认 |
| 项目资产库 | 项目和资产的阶段、负责人、阻断项。 | 继续处理 |
| 治理运营 | 影响交付或客户使用的异常。 | 定位问题 / 前往修复 |
| 平台设置 | 影响生产的配置健康度。 | 处理配置 |

## 6. 状态模型与跳转规则

每个项目/资产需要提供或派生三项统一状态：

- `current_stage`：资料接入、AI 审核、组装 MCP、测试发布、交付确认。
- `blockers`：当前阶段未满足的准入条件，包含优先级、原因、关联对象与修复入口。
- `next_action`：用户可执行的唯一优先动作，包含目标页面、项目/资产上下文和可选焦点记录。

“继续处理”必须携带项目、资产、当前阶段和焦点记录。无论用户从今日待办、资产库、治理运营或交付确认进入，都回到正确的页面、项目与待处理记录，而不是泛化列表。

### 6.1 阶段准入

| 当前阶段 | 进入下一阶段的最低条件 | 不满足时的明确提示 |
| --- | --- | --- |
| 资料接入 | 资料齐全且连接健康。 | 缺哪个资料或哪个连接不健康。 |
| AI 审核 | 高风险候选均已处理。 | 待审核数量、风险原因与审核入口。 |
| 组装 MCP | 必需 Tool 与边界均已确认。 | 缺失 Tool、参数边界或安全规则。 |
| 测试发布 | 测试通过且策略齐全。 | 失败用例、策略缺失与修复入口。 |
| 交付确认 | 交付包、运行说明、验收证据齐全。 | 缺失的具体交付物。 |

后续阶段可以被查看，但相关推进操作不可执行；页面必须展示准入原因、补齐动作和补齐后的结果。

### 6.2 异常回流

治理运营不要求用户理解模块关系，异常直接转换为可执行修复任务：

| 异常 | 引导目标 |
| --- | --- |
| 401 / 403 | 授权或凭证处理。 |
| 参数错误 | 对应 Tool 的边界确认。 |
| 超时 / 5xx | 资料源或接入健康检查。 |
| 验收失败 | 对应资产的测试修复任务。 |

## 7. 空状态与渐进披露

空状态必须提供动作，不展示无解释的空表格：

- 新项目：先上传业务资料。
- 已上传但未识别：提交 AI 识别。
- 没有 MCP 草稿：先确认 Tool 边界。
- 没有发布版本：执行第一次沙箱测试。

侧栏默认展开“资产生产”，其余组折叠。当前处理项目置于侧栏顶部。仅需要用户动作的事项显示待办数字，纯指标不使用红点或数字干扰。

## 8. 交付与技术实施边界

本次页面重构应复用现有功能页面及数据源。优先增加导航分组、上下文路由、阶段进度、行动摘要和下一步动作，再逐步将连续功能改为工作台内的页签/阶段。

实施分三阶段：

1. **导航和待办：** 重组侧栏、实现今日待办、为现有页面补充下一步行动区。
2. **工作台合并：** 将 AI 识别相关页和 MCP 组装相关页分别收敛为页内阶段，同时保留原有操作。
3. **状态闭环：** 引入阶段准入、异常回流、行动型空状态和过程指标。

## 9. 验收与度量

### 9.1 验收标准

1. 管理员侧现有 13 个入口的业务能力均可从新导航进入。
2. 任一进行中的项目或资产都能显示当前阶段、阻断项与下一步动作。
3. 点击“继续处理”能将用户带到正确项目、资产和待处理记录。
4. 每一个生产阶段页面都有单一清晰的优先主按钮。
5. 准入不满足时，用户能看到原因、补齐动作和预期结果。
6. 调用异常可跳转到其对应的修复阶段。
7. 所有空状态都包含可执行的下一步。

### 9.2 效果指标

1. 管理员从登录到开始处理任务的时间低于 30 秒。
2. 资料接入到首个可测试 MCP 的完成率和平均耗时改善。
3. 各阶段无动作停留的项目/资产数量下降。
4. 从监控异常成功跳转到正确修复页面的比例上升。
