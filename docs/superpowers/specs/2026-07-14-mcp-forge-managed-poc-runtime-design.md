# MCP Forge 托管 POC 运行与真实联调设计

**日期：** 2026-07-14  
**目标：** 让一个已完成治理的 MCP 资产能在本机作为标准 SSE MCP 服务运行，并在现有后台完成真实连接、调用监控和交付验收取证。

## 背景与问题

当前后台已完成资料接入、AI 识别、人工确认 Tool 边界、MCP 组成确认、分层审核、测试发布、调用监控和交付管理。已有的 `codegen.js` 也能生成基于 SSE 的独立 MCP Server。

但资产页面中的端点目前是资产元数据；测试发布主要调用 `/admin/simulate-call`。这只能演示平台内的模拟调用，不能证明标准 MCP 客户端能够连接、发现并调用平台生成的 MCP 服务。

## 设计目标

1. 不新增一级导航或独立 POC 页面。
2. 仅允许已经通过既有治理与发布门禁的 MCP 资产启动 POC 运行实例。
3. 使用标准 SSE MCP 协议完成真实的初始化、Tool 发现和 Tool 调用。
4. 将每次真实联调的状态、失败阶段、状态码、耗时和 Trace ID 纳入现有调用监控。
5. 在交付管理中形成可导出的 POC 验收凭证。

## 首个 POC 范围

- 目标资产：`sales_top_products`（销售 TopN 查询）。
- 数据来源：`Mock Connector`。它只用于证明协议、治理与交付链路；页面和交付物必须清楚标注为模拟数据。
- 访问范围：本机 `localhost`，不提供公网地址，不宣称已接入客户生产系统。
- 操作范围：只读 Tool，不覆盖写操作、客户 API、数据库或 SaaS 连接器。

## 方案选择

采用“平台托管本地 SSE 运行实例”。后台使用现有代码生成器为已发布资产生成独立 Node MCP Server，并以受控子进程启动。相比把 MCP 路由混入管理后台，这能保留可独立交付的服务形态；相比人工运行生成包，它能提供平台内的状态管理、测试证据和诊断闭环。

## 页面与交互

### MCP 资产

资产详情增加“POC 运行实例”卡片：

- 显示资产版本、Tool 数、运行状态、SSE 地址、健康状态、最近一次联调时间和 Trace ID。
- 操作包括“启动实例”“停止实例”“查看联调记录”。
- 未发布、未完成验收或存在阻断审核项的资产不可启动，并显示被阻断原因。
- 服务启动后显示 `http://localhost:<port>/sse`，而不是沿用资产元数据中的演示端点。

### 测试发布

同页保留当前能力，但明确分为两类：

- **沙箱预演**：继续使用 `/admin/simulate-call` 预览参数、脱敏与模拟结果；不作为发布或交付证据。
- **真实 MCP 联调**：选择已运行的 POC 实例，依次执行健康检查、MCP 初始化、`tools/list`、`tools/call`。

每个步骤显示成功、失败或未执行。失败时保留失败阶段、HTTP 状态码或协议错误、错误摘要、耗时和 Trace ID，并提供跳转调用监控的入口。

### 调用监控

真实 MCP 运行实例的调用统一写入现有 `platform_call_events`。事件标记来源为 `poc_sse`，记录资产、调用方、状态、耗时、请求摘要、响应摘要、Trace ID 和失败原因。监控页可通过 Trace ID 直接定位联调失败。

### 交付管理

在现有交付包中增加“POC 验收凭证”，至少包括：

- 已验证资产及版本；
- 本机 SSE 地址与健康检查结果；
- 已发现的 Tool 清单；
- 一次真实 Tool 调用的结果摘要、耗时和 Trace ID；
- 执行时间、执行人和 Mock Connector 标识。

## 服务与数据设计

新增运行实例记录，用于持久化资产与本地进程的关联、端口、端点、状态、健康时间、最近错误与最近联调 Trace ID。应用重启后将旧进程记录标记为已停止，避免把无效 PID 展示为运行中。

应用进程内维护受控子进程引用；生成 MCP Server 在收到 Tool 调用后向后台内部事件入口上报调用结果。该入口使用一次运行实例令牌校验，避免任意请求写入监控。MCP 返回内容包含 Trace ID，以便外部 MCP 客户端与后台监控对应。

`Mock Connector` 保持为 Tool 的唯一数据依赖。后续接入客户系统时，以统一 Connector 接口替换其实现，不改变 MCP 协议地址、审核门禁、测试发布和调用监控。

## 状态与验收

运行实例状态：`starting`、`running`、`stopped`、`failed`。

POC 验收必须同时满足以下条件，资产才能显示“POC 已验证”：

1. `GET /health` 返回 HTTP 200。
2. MCP 客户端通过 SSE 完成初始化。
3. MCP 客户端获取到预期 Tool `sales_top_products`。
4. MCP 客户端成功调用该 Tool，且调用监控收到同一 Trace ID 的成功事件。

任一条件失败，验收状态为失败，不覆盖之前的成功记录；页面保留本次失败上下文，便于重新测试和诊断。

## 错误处理

- 子进程启动失败、端口占用、健康检查失败：实例进入 `failed`，显示可读错误，不生成通过凭证。
- MCP 初始化、Tool 发现或 Tool 调用失败：记录具体阶段与协议/HTTP 错误，并产生 Trace ID。
- 未通过治理门禁：拒绝启动，不创建进程。
- 进程异常退出：更新实例状态为 `failed`，保留退出信息；不影响其它资产实例。

## 非目标

- 不实现真实客户 API、数据库或 SaaS Connector。
- 不提供公网部署、多节点编排、客户租户隔离或生产 SLA。
- 不删除现有沙箱预演能力。
- 不将未受审核的 AI 建议直接运行或发布。

## 验证策略

自动化测试覆盖：运行门禁、实例启动与停止、健康检查失败、SSE MCP 的 Tool 发现、成功调用事件写入、失败调用 Trace ID 留存、交付凭证生成。

人工验收覆盖：管理员从已发布资产启动实例，在测试发布完成四步真实联调，在调用监控按 Trace ID 定位记录，并在交付管理中查看 POC 验收凭证。
