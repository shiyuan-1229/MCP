> 结合项目背景、5 个方向分析、科传客户结构（中小型 70%+），提炼为两个落地执行方案
> 更新日期：2026-07-02

---

## 方案选择逻辑

```
全部 5 个方向
  ├─ A·纯 MCP 工厂      → 壁垒太低，不适合作为主方案
  ├─ B·工厂+知识库       → 融入方案一、二中
  ├─ C·企业 AI 能力套件   → 方案二的远期愿景
  ├─ D·行业模板库         → 方案一的核心打法
  └─ E·安全网关平台       → 方案二的核心壁垒

最终提炼为 ↓

  方案一：零售门店 AI 助手（P0 · 优先执行）
  方案二：零售集团 AI 中台（P1 · 长期布局）
```

**科传客户结构（已确认）**：中小型零售连锁占 **70%+** → 方案一为 P0 优先级，暑期 POC 80% 资源投入。

## 二、方案一：零售门店 AI 助手（P0 · 优先执行）

### 2.1 一句话定位

> **帮中小零售商 1 天接入微信 AI，顾客在对话里就能查库存、下单、支付。**

### 2.2 解决什么问题

中小零售连锁（科传 70%+ 客户）面临的困境：
- 想蹭微信 AI 专属卡 / Agent 的流量红利，但没有技术团队
- 系统能跑但没 API，不知道怎么和 AI 对接
- 怕投入太大，想先低成本试水
- 瑞幸已经做了 MCP，再不做自己的门店就被 AI 跳过了

### 2.3 场景流程（端到端）

```
顾客在微信 AI 助手输入：

  "附近有 XX 便利店吗？还有没有元气森林海盐菠萝？"
                    ↓
  AI Agent 调 MCP 门店模板
     ├─ 调「门店查询」Tool → 查找附近 3 家门店
     ├─ 调「库存查询」Tool → 天河北店有 32 箱
     └─ 调「促销查询」Tool → 买 2 箱享 9 折，会员赠 500 积分
                    ↓
  AI 回复：
     "📍天河北店有货，¥9.9/瓶，买 2 箱享 9 折"
                    ↓
  顾客："帮我下单 2 箱，到店自提"
                    ↓
  AI Agent 调 MCP 订单模板
     ├─ 调「创建订单」Tool → 生成订单
     ├─ 调「会员查询」Tool → 使用会员优惠
     └─ 调微信支付 → 用户确认扣款 ✓
                    ↓
  AI 回复：
     "已下单，取货码 6688，到店出示即可取货 🎉"
```

### 2.4 交付内容

| 模块 | 内容 | 交付方式 | 
|------|------|---------|
| **MCP 模板库（核心）** | 预装 5 大零售模块：商品搜索/库存查询/门店列表/会员积分/订单创建 | 标准化产品 |
| **知识库** | 导入商品目录、促销规则、常见 FAQ，做成 MCP Resources | 半标准化配置 |
| **微信 Agent 对接** | 配置 MCP Server 地址，连接微信 AI 专属卡支付 | 标准化配置 |
| **数据源适配** | 连接客户已有系统的库存/订单数据（支持 Excel/简易 API/手动录入） | 轻量配置 |

### 2.5 收费模型

| 收费项 | 模式 | 金额 | 说明 |
|-------|------|------|------|
| **平台注册费** | 按门店年费 | ¥3,000-¥5,000/门店/年 | 覆盖平台和模板成本 |
| **MCP 调用费** | 按量计费 | ¥0.2-¥0.8/千次 | 弹性部分，与业务增长正相关 |
| **知识库搭建** | 一次性项目 | ¥10,000-¥30,000 | 商品目录/促销规则整理导入 |
| **30 门店客户年贡献** | **¥15-25 万/年** | | 注册费 ¥9-15 万 + 知识库 ¥1-3 万 + 调用费 ¥5-7 万 |

### 2.6 POC 规划（8 周）

```
第 1-2 周 ── 模板开发（5 个核心 MCP 模板：商品/库存/门店/会员/订单）
第 3-4 周 ── 知识库对接 + 微信支付沙箱打通
第 5-6 周 ── 找 1 个真实便利店完成端到端验证
第 7-8 周 ── 产出案例数据 + 签约材料 + 市场推广计划
```

### 2.7 为什么这个方案最适合科传

1. **客户匹配度最高**：科传 70%+ 客户是中小零售连锁，需求高度同质化
2. **上手最快**：模板化交付，1 天上线，客户决策成本低
3. **可规模化**：模板→复制→拓展，边际成本趋近于零，适合铺量
4. **与微信 AI 支付完美配合**：查库存→下单→支付闭环，所有环节都在微信里完成

## 三、方案二：零售集团 AI 中台（P1 · 长期布局）

### 3.1 一句话定位

> **帮大型零售集团把 ERP/POS/CRM 等旧系统整合成统一的 AI 中台，安全可控地接入微信 Agent。**

### 3.2 解决什么问题

大型零售集团（科传 20-30% 客户）的困境：
- 系统太老太多（ERP、CRM、POS、WMS 各一套），彼此不互通
- 想把业务能力开放给 AI，但 IT 部门最担心安全问题——"API 暴露了谁负责？"
- 需要整体方案，不想自己拼凑

### 3.3 场景流程（端到端）

```
顾客在微信 AI 助手中输入：

  "附近商场有什么店在打折？"
                    ↓
  AI Agent 调 MCP 中台（经过 Forge Shield 安全网关）
     ├─ 认证：验证调用身份合法
     ├─ 调「门店促销」Tool → 查商场各店铺活动
     └─ 权限校验：只能查公开促销信息，不暴露商户后台
                    ↓
  AI 回复：
     "A 区 ZARA 全场 7 折，B 区星巴克买一送一"
                    ↓
  顾客："用我会员积分兑换一张星巴克券"
                    ↓
  AI Agent 再次调 MCP 中台
     ├─ 脱敏处理：查询会员信息时自动隐藏手机号
     ├─ 调「会员积分」Tool → 查积分余额
     ├─ 调「营销核销」Tool → 兑换优惠券
     └─ 记录审计日志：某时某分，AI 为某会员兑换了星巴克券
                    ↓
  AI 回复：
     "已为您兑换，星巴克买一送一券已推送至微信卡包 🎫"
```

### 3.4 交付内容

| 模块 | 内容 | 交付方式 |
|------|------|---------|
| **旧系统适配器** | 给客户老系统（ERP/CRM/POS）加一层标准化 API 层 | 定制开发 |
| **MCP 模板库** | 按客户业务定制 15-30 个 MCP Tool | 半定制 |
| **Forge Shield 安全网关** | 身份认证 + 权限校验 + 数据脱敏 + 审计日志 + 限流熔断 | 标准化产品 |
| **知识库** | 商品知识、会员权益、促销规则、运营手册 | 半定制 |
| **Token 管理与计费** | 统一管控 API 调用成本，生成内部结算报表 | 标准化产品 |

### 3.5 收费模型

| 收费项 | 模式 | 金额 | 说明 |
|-------|------|------|------|
| **旧系统整合费** | 一次性项目 | ¥80,000-¥200,000 | 适配器开发 + 系统对接 |
| **中台平台年费** | 年费 | ¥80,000-¥200,000/年 | MCP 模板 + 安全网关 + Token 管理 |
| **MCP 调用费** | 按量 | ¥0.5-¥2/万次 | 弹性计费 |
| **知识库搭建** | 一次性项目 | ¥30,000-¥80,000 | 文档整理 + 知识库建设 |
| **典型集团客户首年** | **¥30-50 万** | | 整合费 ¥10-20 万 + 年费 ¥10-20 万 + 知识库 ¥3-8 万 + 调用费弹性 |

### 3.6 POC 规划（16 周）

```
第 1-4 周 ── 客户旧系统调研 → 适配器设计 → 安全网关基础搭建
第 5-8 周 ── 15+ MCP Tool 开发 → 知识库导入 → 全线系统联调
第 9-12 周 ── 接入微信 Agent → 跑通"查→推→下→付"完整闭环
第 13-16 周 ── 安全审计 → 性能压测 → 产出标杆案例 → 准备签约材料
```

### 3.7 暑期 POC 投入建议

> 鉴于方案二周期较长（16 周），暑假如果时间不够，建议只做**技术预研 + 1 个意向客户的前期调研**，签约和交付放在秋季。

**暑期可做的最小范围**：
1. 调研 1-2 个大型客户的旧系统现状，产出《旧系统适配可行性报告》（4 周）
2. 完成 Forge Shield 安全网关的核心功能原型（认证 + 审计日志）（6 周）
3. 选一个客户最简单的 3-5 个 API 完成端到端联调，验证可行性（6 周）

## 四、两个方案的关系与配合

```
                   科传客户矩阵
        ┌────────────────────────────────┐
        │   70%+ 中小型连锁              │  20-30% 大型集团
        │                                │
        │  ┌─────────────────────┐  ┌─────────────────────┐
        │  │  方案一 · P0 优先    │  │  方案二 · P1 备选    │
        │  │  零售门店 AI 助手    │──│  零售集团 AI 中台    │
        │  │  模板化 · 快速铺量   │  │  定制化 · 深度绑定   │
        │  └─────────┬───────────┘  └──────────┬──────────┘
        │            │                          │
        │            └────── 共享底层 ──────┘
        │          MCP 引擎 + 知识库 + 微信支付
        └────────────────────────────────────────┘
```

**两条线的关系**：
- **技术同源**：共享 MCP 引擎、知识库模块、微信 Agent 对接层
- **交付分层**：方案一模板化（可复制），方案二定制化（深绑定）
- **客户升级**：小客户用方案一入门，成长后升级到方案二

## 五、下周会议讨论建议

### 5.1 建议讨论顺序

1. **对齐大方向**（10 分钟）
   - 方案一（门店 AI 助手）vs 方案二（集团 AI 中台）
   - 是否认可"方案一 P0、方案二 P1"的优先级？

2. **敲定 POC 范围**（15 分钟）
   - 暑期做方案一的 8 周 POC？还是方案二的 16 周？
   - 建议：暑假做方案一 POC（8 周），方案二只做技术预研

3. **定盈利模式**（10 分钟）
   - 注册制 + 调用制的比例？
   - 初期是否用"注册费打折，调用费后付"获客？

4. **分资源和责任**（5 分钟）
   - 谁做模板开发？谁做知识库？谁对接微信支付？
   - 需要 1 产品 + 1 后端 + 1 前端投入方案一 POC

### 5.2 方案选择建议

> **如果是暑期 8 周 POC → 方案一（门店 AI 助手）是唯一现实的选择**
> 方案二建议秋季再启动，暑期先做技术预研
