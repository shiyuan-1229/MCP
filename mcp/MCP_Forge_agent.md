# MCP Forge 管理后台重构方案：从「运维接入台」到「MCP 资产生成工厂」

> 生成日期：2026-07-09
> 状态：架构草案

---

## 一、产品重新定位（一句话）

本后台是一个「企业业务资料 → MCP 资产」的 AI 加工工厂控制台：客户提交接口文档/数据库结构/业务说明，平台自动识别接口、生成 OpenAPI 规范、映射为 MCP Tools、封装成 MCP Server，并提供测试、发布、交付和持续治理的全流程管理。

核心动词链：接收 → 识别 → 规范化 → 映射 → 封装 → 验证 → 交付 → 治理

| 序号 | 旧导航名 | 定位偏移问题 |
|------|---------|------------|
| 1 | 项目工作台 | 太泛，像通用 dashboard |
| 2 | MCP 工厂 | 方向对，但内部子页面顺序需要重排 |
| 3 | 知识库 | 与核心流程关系弱 |
| 4 | 接入配置 | 最大的误导来源，暗示"客户自己接入" |
| 5 | 测试与发布 | 方向对 |
| 6 | 网关与治理 | 方向对，但位置太靠前 |
| 7 | 使用统计 | 方向对，但应后移 |
| 8 | 计费管理 | 辅助能力 |
| 9 | 文件下载 | 太弱，应升级为"交付管理" |

### 新一级导航（推荐）

| 序号 | 新导航名 | 核心职责 | 对应旧模块 |
|------|---------|---------|-----------|
| 1 | 生成总览 | 全局资产生成看板、进度追踪、最近动态 | 项目工作台（重做） |
| 2 | 资料接入 | 客户业务资料的接收、登记、预处理 | MCP工厂"数据源接入" + 接入配置（拆分） |
| 3 | 接口识别 | AI 识别接口、字段、参数，产出 OpenAPI 草稿 | MCP工厂"OpenAPI 识别结果" |
| 4 | Tool 映射 | OpenAPI → MCP Tools 的映射确认与安全处理 | MCP工厂"Tool 映射关系" + "安全处理预览" |
| 5 | MCP 资产 | 已封装的 MCP Server 列表、详情、状态管理 | MCP工厂"MCP 资产列表" + "生成流程时间线" |
| 6 | 测试发布 | 沙箱试调、发布管理、版本控制 | 测试与发布（保留微调） |
| 7 | 交付管理 | 交付物打包、下载、交付确认 | 文件下载（升级） |
| 8 | 治理与统计 | 网关策略、调用监控、审计日志、用量分析 | 网关与治理 + 使用统计（合并） |
| 9 | 设置 | 客户管理、项目配置、计费、知识库、API 凭证 | 计费管理 + 知识库 + 接入配置（凭证部分） |

设计原则：前 7 个是主流程，第 8 个是运营支撑，第 9 个是系统设置。

## 三、逐项模块映射

### 1. 项目工作台
- 建议改名：生成总览
- 是否保留：保留框架，重做内容
- 在新产品逻辑里的职责：展示全平台 MCP 资产生产线的总进度——多少资料已接入、多少接口已识别、多少 OpenAPI 已确认、多少 Tools 已映射、多少 Server 已封装、多少已发布、多少已交付。是厂长视角的驾驶舱。
- 是否应该作为一级导航：是，且为首页

### 2. MCP 工厂
- 建议改名：拆分为资料接入/接口识别/Tool映射/MCP资产四个一级导航
- 是否保留：保留子页面能力，取消作为一级容器
- 职责：原来的工厂概念正确，但把工厂拆成流水线的各个工位
- 是否应该作为一级导航：否，拆分后不再需要

### 3. MCP 工厂 - 数据源接入
- 建议改名：资料接入（提升为一级导航）
- 是否保留：是
- 职责：客户业务材料的接收入口——上传 Swagger/Postman 文件、粘贴 API 文档链接、上传数据库 DDL、上传字段说明 Excel/业务文档 PDF
- 是否应该作为一级导航：是，主流程第一步

### 4. MCP 工厂 - OpenAPI 识别结果
- 建议改名：接口识别（提升为一级导航）
- 是否保留：是
- 职责：AI 从业务资料中识别出接口列表、请求方法、参数结构、数据模型，生成 OpenAPI 3.0 规范草案
- 是否应该作为一级导航：是，主流程第二步

### 5. MCP 工厂 - Tool 映射关系
- 建议改名：Tool 映射（提升为一级导航）
- 是否保留：是
- 职责：将确认后的 OpenAPI endpoints 映射为 MCP Tools——命名、描述、参数 schema、返回格式
- 是否应该作为一级导航：是，主流程第三步

### 6. MCP 工厂 - 安全处理预览
- 建议改名：降级为 Tool 映射的子 Tab
- 是否保留：是，但不作为独立页面
- 职责：敏感字段脱敏、鉴权注入策略、输入校验规则
- 是否应该作为一级导航：否

### 7. MCP 工厂 - 生成流程时间线
- 建议改名：降级为 MCP 资产详情页内的 Tab
- 是否保留：是，但不作为独立页面
- 职责：每个 MCP Server 资产详情页中展示其生成历程
- 是否应该作为一级导航：否

### 8. MCP 工厂 - MCP 资产列表
- 建议改名：MCP 资产（提升为一级导航）
- 是否保留：是
- 职责：已封装完成的 MCP Server 列表，展示状态、Tools 数量、关联客户/项目
- 是否应该作为一级导航：是

### 9. 知识库
- 建议改名：降级到设置的子页面
- 是否保留：是，但后移
- 职责：存放业务术语表、行业知识库、历史生成参考
- 是否应该作为一级导航：否

### 10. 接入配置
- 建议改名：拆分——凭证管理部分降入设置，健康检查/审计/Webhook 降入治理与统计
- 是否保留：部分保留，取消一级导航
- 职责：改为平台侧管理 MCP Server 的访问凭证和部署配置
- 是否应该作为一级导航：否——这是最大的误导点

### 11. 接入配置 - API Key / OAuth / SSE / mTLS / Webhook
- 建议改名：凭证与协议 → 降入设置
- 是否保留：是
- 职责：MCP Server 发布后的访问凭证管理
- 是否应该作为一级导航：否

### 12. 接入配置 - 客户接入配置
- 建议改名：删除或重命名为客户连接配置 → 降入设置
- 是否保留：重构
- 职责：改为 MCP 资产详情页内的"连接信息"Tab
- 是否应该作为一级导航：否

### 13. 接入配置 - 健康检查 / 审计历史 / Webhook 日志
- 建议改名：降入治理与统计
- 是否保留：是
- 职责：运营监控能力
- 是否应该作为一级导航：否

### 14. 测试与发布
- 建议改名：测试发布（合并命名，微调）
- 是否保留：是
- 职责：沙箱试调、版本发布管理、回滚
- 是否应该作为一级导航：是

### 15. 文件下载 / 交付物工作台
- 建议改名：交付管理（升级为一级导航）
- 是否保留：是，并增强
- 职责：打包 MCP Server、生成交付报告、记录交付确认、跟踪客户验收状态
- 是否应该作为一级导航：是

- 建议改名：合并进治理与统计
- 是否保留：是
- 职责：限流策略、调用鉴权、安全审计
- 是否应该作为一级导航：作为治理与统计的子 Tab

### 17. 使用统计
- 建议改名：合并进治理与统计
- 是否保留：是
- 职责：调用量、成功率、Top Tools、客户维度统计
- 是否应该作为一级导航：作为治理与统计的子 Tab

- 是否应该作为一级导航：否

## 四、后台主流程

客户业务资料 → 资料接入 → 接口识别 → Tool映射 → MCP资产 → 测试发布 → 交付管理
                                                                   ↑
                                                        治理与统计（横向支撑）
                                                        设置（底层配置）

叙事主线：客户给料 → AI 识别 → 人工确认 → 封装成器 → 测试 → 交付 → 持续监控

## 五、页面优先级分类

### 最值得保留（核心价值页）
- 资料接入（原数据源接入）：流程入口
- 接口识别（原OpenAPI识别结果）：AI 核心能力展示
- Tool 映射（原Tool映射关系）：转换关键节点
- MCP 资产列表：产品产出物管理
- 沙箱试调：能力验证
- 交付物工作台：闭环终点

### 应该弱化（降级处理）
- 知识库 → 降为设置的子页面
- 计费管理 → 降为设置的子页面
- API Key/OAuth/SSE/mTLS 配置 → 降为设置子页面
- 安全处理预览 → 降为 Tool 映射的子 Tab

### 应该后移（改变位置）
- 接入配置：一级导航 → 拆分降级
- 网关与治理：一级导航 → 合并为治理与统计
- 使用统计：一级导航 → 合并为治理与统计
- 客户接入配置：子页面 → MCP 资产详情页内

### 应该改为辅助能力
- 健康检查 → 治理与统计的监控 Tab
- 审计历史 → 治理与统计的审计 Tab
- Webhook 日志 → 治理与统计的日志 Tab
- 生成流程时间线 → MCP 资产详情页的历程 Tab

## 六、新首页方案：生成总览

### 定位
不是"项目工作台"，而是资产生产线的厂长驾驶舱。

### 页面结构

① 顶部：资产生成漏斗（核心指标卡）
- 已接入资料 → 已识别接口 → 已确认 OpenAPI → 已映射 Tools → 已封装 Server → 已发布

② 左侧：资产状态分布
- 待识别、待确认 OpenAPI、待映射 Tools、待封装、待测试、待发布、待交付

③ 右侧：最近生成动态（时间线）

④ 底部：本周产能（趋势图）

关键设计点：首页不出现技术配置概念，不出现零售 Demo 业务词汇。

## 九、实施建议

改动量评估：不需要重写后端 API。现有表结构完全能支撑新架构。

主要改动：
1. 前端导航重构（app.js / state.js 中的路由和菜单）
2. 首页内容重做（从项目工作台改为生成总览指标卡）
3. 文案替换（导航名、页面标题、按钮文字）
4. MCP 工厂拆分（6个子页面拆到4个一级导航下）
5. 接入配置降级（拆分到设置和治理与统计）

现有渲染逻辑（renderers.js）和数据接口（server.js API）大部分可复用。
