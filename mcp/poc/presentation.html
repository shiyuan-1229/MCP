企业落地 AI 的四大痛点

🧩

系统对接碎片化

每个业务系统接口协议、认证方式、返回格式都不一样，AI 团队重复造轮子。

🔐

权限与数据安全难控

大模型直接访问数据库风险高，缺少租户隔离、字段脱敏、调用审计。

🚀

从 POC 到生产跨度大

沙箱跑通不等于可交付，缺版本管理、灰度发布、效果追踪和计费闭环。

🏭

多客户复用成本高

每个客户都定制开发，能力无法沉淀为可复用资产，交付人效低。

业务价值
MCP Forge 的定位不是做一个 Chatbot，而是帮助企业把自有能力封装成 AI 可调用的标准化资产，并治理这些资产的全生命周期。

平台架构：从数据源到客户交付

🔌

数据源接入

REST API、Database、知识库等多种数据源统一接入，支持 API Key / JWT / OAuth / mTLS 等认证模式。

🏭

MCP 工厂

把业务能力定义为 MCP 资产：名称、能力描述、版本、端点、工具列表，支持草稿 → 测试 → 发布全流程。

🛡️

网关与治理

按客户/项目配置认证授权、限流策略、字段脱敏规则，审计每一次调用。

🧪

测试与发布

内置沙箱试调，记录版本发布历史，支持测试通过后再正式上线。

💰

计费管理

实施费、年费/能力包、按调用量/效果计费，形成商业闭环。

🚪

客户交付门户

客户登录后只看到自有项目、MCP 资产、调用效果、交付文件下载。

案例一

美佳零售集团

连锁零售企业，需要让门店店长、会员运营、客服都能通过 AI 快速获取经营数据和服务知识。

  • 销售 TopN 查询：AI 直接问“本月最好卖的 10 个 SKU”
  • 会员权益提醒：自动识别积分过期、券到期并推送
  • 门店知识库：服务手册、退换货政策实时问答

案例二

华智制造

离散制造企业，生产工单、质检记录分散在 MES 和 QMS 中，管理层需要快速掌握产线状态。

  • 工单查询：按产线、工序、状态查询在制工单
  • 质检分析：统计不良率、定位异常批次
  • 设备维护提醒：结合工单进度推送保养建议

项目状态

进度：62%

quality_inspection（v0.3.0 草稿）

MCP 工厂：能力定义与资产生产

📦

MCP 资产列表

名称、能力描述、状态、版本、所属项目一目了然。

🔌

数据源接入

每个资产关联哪些数据源、什么协议、认证方式、当前连接状态。

➕

新建草稿

点击按钮即可创建新 MCP 资产草稿，自动分配版本 v0.1.0。

关键设计：MCP 资产是“业务能力的标准化封装”，不是一次性接口。同一个能力可以被多个客户、多个 Agent 复用。

测试与发布：沙箱试调

发布记录

记录每个 MCP 资产的版本、状态、测试时间、发布时间、发布说明。

示例：

  • sales_top_products v1.2.0：已发布，新增门店筛选和 TopN 参数
  • member_expiring_benefits v1.1.0：已发布，新增过期权益提醒
  • store_service_kb v0.9.0：测试通过，等待客户发布窗口

沙箱试调

选择工具 → 运行试调 → 查看结果

sales_top_products:
{ date_range: "month", top_n: 8 }

member_expiring_benefits:
{ member_id: "M10001" }

结果会记录到 call_events，
用于统计调用量和成功率。

网关与治理：安全可控

🔑

认证与授权

支持 API Key、JWT、OAuth、mTLS 等多种认证模式，按项目和调用方限定授权范围。

⏱️

限流策略

可按客户、项目、应用设置 RPM，如零售 600 rpm、金融 60 rpm、制造 120 rpm。

🎭

字段脱敏

手机号、会员 ID、订单金额、身份证号、工号等敏感字段可按策略自动脱敏。

📋

调用审计

每次调用记录 caller、trace_id、状态、耗时、业务结果，满足合规要求。

使用统计：效果可观测

核心指标

  • 调用量：按调用事件统计
  • 成功率：成功 / 总调用
  • 平均响应：按日志统计毫秒数
  • 效果事件：领券、核销、转化等业务动作
