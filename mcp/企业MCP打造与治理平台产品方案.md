# 企业 MCP 打造与治理平台产品方案

> 本文档作为 MCP Forge 项目的产品总纲，用于统一后续方案文档、演示脚本、后台改版、销售话术和交付边界。

---

## 一、项目定位

### 1.1 一句话定义

企业 MCP 打造与治理平台，是一套帮助企业把已有系统、接口、数据和知识资料封装成 AI 可调用 MCP 工具的技术服务平台。

平台不替客户重做业务系统，也不替客户建设新的经营分析平台，而是把客户已经存在的业务能力，通过 MCP 的方式安全、可控、可追溯地开放给企业微信、OA、智能客服、小程序、App、AI Agent 等入口调用。

### 1.2 核心判断

企业未来会有越来越多业务请求从“人点系统页面”变成“人问 AI 助手”。如果企业已有的 ERP、POS、CRM、会员、营销、知识库、工单等系统不能被 AI 调用，这些系统就会变成信息孤岛。

MCP Forge 的价值不在于做一个单纯的代码生成器，而在于帮助企业完成三件事：

1. 把已有业务系统变成 AI 可调用能力。
2. 把这些能力纳入认证、授权、审计和网关治理。
3. 用调用数据和业务转化证明 MCP 已交付、有人用、有效果。

## 二、我们提供什么

### 2.1 MCP 能力生成

平台支持将企业已有接口、数据库、文档资料和标准业务模板封装成 MCP 工具，包括：

- 从 OpenAPI、Swagger、REST API 生成 MCP Tool。
- 从数据库查询、报表接口、业务中台接口生成 MCP Tool。
- 从商品资料、会员规则、服务手册、FAQ 等内容生成 MCP Resource 或知识库检索能力。
- 基于零售、商业地产、物业、制造等行业场景提供预置模板。

生成后的 MCP 能力可以被企业微信、OA、智能客服、AI Agent、小程序和 App 调用。

### 2.2 MCP 生命周期管理

平台不只负责生成 MCP，还负责从配置到上线的完整生命周期：

- 数据源接入。
- MCP 定义生成。
- 参数与返回结构配置。
- 沙箱测试。
- 发布审批。
- 生产发布。
- 版本管理。
- 下线与回滚。
- 配置文件导出。

### 2.3 认证、授权与安全治理

企业系统不能直接裸露给 AI 调用，因此平台提供 MCP 网关与治理能力：

- 认证：支持 API Key、OAuth、JWT、企业内部 Token 等方式。
- 授权：控制不同 Agent、用户、部门、场景可调用哪些 MCP。
- 限流：按客户、应用、接口、时间窗口限制调用频率。
- 审计：记录谁在什么时间调用了哪个 MCP、传了什么参数、返回了什么结果摘要。
- 脱敏：对手机号、会员号、订单号、金额、身份信息等敏感字段做规则化脱敏。
- 追踪：为每次调用生成 trace_id，便于问题排查和责任追溯。

### 2.4 使用统计与效果证明

平台提供 MCP 使用数据，让客户知道这些能力是否真正被使用：

- 调用量。
- 调用成功率。
- 平均响应时间。
- 错误率。
- 高频问题。
- 高频 MCP。
- 领券数。
- 核销数。
- 转化数。
- 可导出的调用日志和效果报告。

这里的重点不是替客户做业务分析，而是证明 MCP 的交付效果。

## 三、客户得到什么

### 3.1 对业务方

业务方得到的是“可被 AI 调用的业务能力”，而不是一个新的业务系统。

例如：

- 店总可以在企微或 OA 里问销售 Top10 商品。
- 运营人员可以问某个活动的领券和核销情况。
- 客服可以让 AI 查询会员积分、优惠券、订单状态。
- 消费者可以在小程序或客服入口中查询自己的权益。

### 3.2 对 IT 方

IT 方得到的是一套可控的 AI 接入层。

企业不用把内部系统直接暴露给大模型或第三方 Agent，而是通过 MCP 网关统一控制：

- 哪些系统可以被调用。
- 哪些数据可以返回。
- 哪些用户有权限调用。
- 每次调用是否有日志。
- 调用异常是否能追踪。
- 敏感字段是否被脱敏。

### 3.3 对管理层

管理层得到的是一组清晰的交付资产和效果数据：

- 已交付多少个 MCP。
- 每个 MCP 覆盖什么业务能力。
- 哪些 MCP 已发布，哪些还在测试。
- 每天有多少调用。
- 调用是否稳定。
- 是否带来领券、核销、转化等可见效果。
- 这些能力对应多少费用。

## 四、两个典型场景

### 4.1 场景一：企业内部经营查询

**用户角色**：店总、区域经理、运营人员、总部管理人员。

**使用入口**：企业微信、OA、内部智能机器人。

**典型问题**：

> “这个月销售 Top10 是什么商品？”

**调用链路**：

1. 用户在企微或 OA 里向智能机器人提问。
2. 智能机器人识别为经营查询意图。
3. 机器人调用 MCP Forge 发布的 `sales_top_products` 能力。
4. MCP 网关完成身份认证、权限校验和审计记录。
5. MCP Tool 查询客户已有 POS、ERP 或销售报表接口。
6. 返回 TopN 商品、销售额、销量、门店或时间范围。
7. AI 以自然语言返回结果。

**平台价值**：

客户不需要在 MCP Forge 里看完整销售报表，也不需要把销售系统迁移过来。MCP Forge 只负责把“查询销售 TopN”这个已有业务能力安全地暴露给 AI。

### 4.2 场景二：消费者会员权益查询

**用户角色**：消费者、会员、客服人员。

**使用入口**：小程序、App、智能客服、微信公众号、企业客服机器人。

**典型问题**：

> “我现在有多少积分？有没有快到期的？”

**调用链路**：

1. 消费者在客服或小程序中提问。
2. AI 识别为会员权益查询。
3. AI 调用 MCP Forge 发布的 `member_expiring_benefits` 能力。
4. MCP 网关校验会员身份和调用权限。
5. MCP Tool 查询客户已有 CRM、会员系统、优惠券系统。
6. 返回当前积分、即将到期积分、可用优惠券、即将到期优惠券。
7. AI 给出自然语言提示，并可引导领券或兑换。

**平台价值**：

客户不用单独建设一个新的会员查询平台，也不用让消费者理解复杂的会员规则。MCP Forge 只是把客户已有会员能力封装成 AI 可调用服务，并通过调用量、领券、核销等数据证明价值。

## 五、后台分角色怎么看

### 5.1 科传侧后台

科传侧后台用于管理客户、项目和 MCP 交付全过程，核心视角是“客户项目资产”。

科传侧应重点看到：

- 企业客户列表。
- 每个客户下的项目。
- 每个项目下的数据源。
- 每个项目生成的 MCP 能力。
- MCP 状态：草稿、测试中、可发布、已发布、已下线。
- MCP 覆盖能力：经营查询、会员权益、优惠券、知识库、工单、导览等。
- 安全配置：认证方式、授权范围、限流策略、脱敏规则。
- 调用审计：调用方、调用时间、调用结果、trace_id。
- 交付文件：配置包、接口文档、测试报告、上线报告。
- 计费信息：实施费、年费、调用量、增值费用。

科传侧后台不应以库存、订单、商品运营、销售报表为主线。这些只是客户已有系统中的业务数据，不是平台要替客户经营的内容。

### 5.2 客户侧后台

客户侧后台用于查看 MCP 资产和使用效果，重点是“我买到了什么、上线了什么、有没有效果”。

客户侧应重点看到：

- 已交付的 MCP 资产。
- 每个 MCP 覆盖的业务能力。
- 发布状态。
- 最近调用量。
- 调用成功率。
- 平均响应时间。
- 业务效果指标。
- 可下载的配置文件、调用日志、效果报告。

客户侧不需要在本平台查看完整库存、订单、会员、商品、销售等业务操作页面。那些仍然应该在客户自己的专业系统里完成。

### 5.3 终端用户侧

终端用户不直接登录 MCP Forge 后台。

他们只通过企业微信、OA、智能客服、小程序、App 等入口使用 AI 问答能力。MCP Forge 在背后提供工具调用、安全治理和审计追踪。

## 六、怎么交付

### 6.1 标准交付流程

1. **需求确认**
   - 明确客户希望 AI 能调用哪些业务能力。
   - 明确使用入口：企微、OA、智能客服、小程序或 App。
   - 明确首批场景，优先选择 2 到 3 个简单高频问题。

2. **系统与数据源梳理**
   - 梳理客户已有 ERP、POS、CRM、会员、优惠券、知识库等系统。
   - 确认可用接口、认证方式、字段结构和数据权限。
   - 对没有接口的资料，整理为知识库或半结构化数据。

3. **MCP 能力设计**
   - 定义 MCP Tool 名称、入参、出参、权限范围。
   - 设计最小可用能力，不追求一次性覆盖所有业务。
   - 明确哪些能力需要脱敏、审批或人工确认。

4. **MCP 生成与配置**
   - 使用平台生成 MCP 定义和服务。
   - 配置数据源、认证方式、授权规则和限流策略。
   - 配置知识库检索或业务规则。

5. **沙箱测试**
   - 用真实问题测试 MCP 调用。
   - 验证返回内容是否准确。
   - 验证权限、脱敏、审计日志是否生效。

6. **发布上线**
   - 发布到生产 MCP 网关。
   - 接入客户指定 AI 入口。
   - 交付配置文件、接口说明、测试报告和上线记录。

7. **运营与复盘**
   - 观察调用量、成功率、错误率。
   - 输出使用效果报告。
   - 根据调用情况追加新 MCP 能力。

### 6.2 首期建议交付范围

首期不要做大而全平台，建议围绕两个示例能力跑通闭环：

- 企业内部：销售 TopN 查询 MCP。
- 消费者服务：会员积分与到期权益查询 MCP。

这两个能力可以证明：

- AI 能调客户系统。
- MCP 能被安全治理。
- 客户后台能看到资产和调用效果。
- 项目可以按实施、能力包和调用量收费。

## 七、怎么收费

### 7.1 收费原则

客户为“业务能力接入 AI”付费，而不是为一个新的业务看板付费。

收费应围绕交付成本、能力资产和持续调用价值设计。

### 7.2 收费结构

| 收费项 | 收费方式 | 说明 |
|---|---|---|
| 实施费 | 一次性项目费 | 覆盖需求梳理、系统对接、MCP 配置、知识库整理、测试上线 |
| MCP 能力包 | 按能力或业务域收费 | 如销售查询、会员权益、优惠券、知识库问答、工单查询 |
| 平台年费 | 按客户、项目或 MCP 数量收费 | 覆盖托管、监控、网关、权限、审计、报表 |
| 调用量费用 | 按调用次数阶梯计费 | 与实际使用量挂钩，适合上线后持续收费 |
| 效果增值费 | 可选 | 可按领券、核销、转化等营销效果设计增值收费 |
| 私有化部署费 | 可选 | 面向安全要求高的大型客户 |

### 7.3 推荐打包方式

**基础版**

- 适合首期试点客户。
- 包含 2 到 3 个 MCP 能力。
- 包含基础认证、调用统计、配置导出。
- 收费以实施费加基础年费为主。

**专业版**

- 适合有多个业务域的企业客户。
- 包含多个 MCP 能力包。
- 包含知识库、测试发布、权限控制、调用审计、基础网关。
- 收费以实施费、能力包年费和调用量计费组合为主。

**企业版**

- 适合大型集团客户。
- 包含私有化部署、安全网关、复杂权限、脱敏策略、审计报表、专属支持。
- 收费以项目实施费、平台年费、调用量和运维服务费组合为主。

## 八、不做什么

### 8.1 不做客户的业务经营系统

平台不替代客户已有的 ERP、POS、CRM、WMS、会员系统、优惠券系统、BI 系统。

库存管理、商品管理、订单处理、会员运营、销售报表等业务操作，仍然应在客户自己的专业系统中完成。

### 8.2 不做大而全的数据分析平台

平台可以展示 MCP 调用量、成功率、响应时间、领券、核销、转化等效果指标，但不做完整 BI，不做复杂经营分析，不做客户业务决策模型。

销售 TopN、积分查询、优惠券查询只是 MCP 调用示例，不应扩展成完整销售分析系统或会员运营系统。

### 8.3 不让客户自己研究 MCP 协议

客户不需要理解 MCP 协议细节，不需要自己写 MCP Server，不需要自己配置复杂工具描述。

客户提供业务系统接口、数据资料和权限规则，平台负责生成、测试、发布和治理。

### 8.4 不承诺一次性覆盖所有系统

首期交付应选择高频、低风险、价值清晰的能力。

优先跑通 2 到 3 个场景，再逐步扩展更多 MCP。避免一开始就接入所有系统、所有接口、所有业务域。

## 九、对现有项目的统一要求

后续所有文档、原型和演示材料应统一遵循以下表达：

1. 产品名称优先使用“企业 MCP 打造与治理平台”。
2. 零售门店、会员积分、销售 TopN 是示例场景，不是产品边界。
3. 后台主线是客户、项目、MCP 资产、发布、网关、统计、计费。
4. 商品、库存、订单、销售等页面应作为 Demo 数据源或示例，不作为正式产品主导航。
5. 销售话术重点是“帮客户把已有业务能力接入 AI，并安全可控地证明使用效果”。

---

## 十、总结

企业 MCP 打造与治理平台的核心价值是：让企业已有业务能力被 AI 安全调用。

平台交付的不是一套新的业务系统，而是一组可被 AI 调用、可被权限控制、可被审计追踪、可被效果证明的 MCP 能力资产。

客户最终购买的是：

> 把我的系统接入 AI 的能力，以及这套能力可控、可用、可证明、可持续运营的服务。
