## 一、开场（30 秒，必背）

> **单句版**（最常用）：
>
> "我做了一个 **MCP Forge** —— 它把零售商的旧系统**自动变成 AI 能调用的接口**，
> 让顾客在微信对话框里**一句话查库存、查优惠、下单**。"

**这一句话覆盖的三个关键词**：
- **MCP** — 协议层，让 AI 能调用工具（区别于传统 LLM 只会聊天）
- **Forge** — 锻造，暗示把"原料（旧系统）"加工成"MCP 接口"
- **微信 AI** — 入口，绑定腾讯生态（校招面试官会很有感）

## 二、问题被问到时怎么答（高频 Q&A）

### Q1：你这个项目解决什么问题？

> 答：**"传统零售商系统老、数据出不来，老板想接入微信 AI 但不知道怎么做。"**
>
> - 顾客侧：晚上想买东西打电话没人接；店员查个库存要翻半小时
> - 商家侧：促销靠店员嘴说，会员积分过期都没人知道
> - 行业侧：瑞幸花了几百万做了 MCP，中小零售商根本做不起

### Q2：MCP 是什么？为什么用 MCP 不用 REST？

> 答：**"MCP = Model Context Protocol，2024 年 11 月 Anthropic 发布的 AI 工具调用协议。"**
>
> - **REST**：给人用的，要看文档、写代码才能调
> - **MCP**：给 AI 用的，AI 自己看 Tool 描述就能决定调哪个、传什么参数
> - 比喻：**MCP 是 AI 时代的 USB**，每个 Tool 就像一个 U 盘，插上就能用
> - 对腾讯的价值：**元器 / 微信 AI 全部支持 MCP 插件**（已有 24 个内置工具）

### Q3：技术栈和实现？

> 答：分 4 层背——
>
> | 层 | 技术 | 用途 |
> |----|------|------|
> | MCP Server 运行时 | Node.js 22 + @modelcontextprotocol/sdk | 暴露 13 个 Tool 给 AI |
> | 数据层 | SQLite + better-sqlite3 | 模拟零售业务数据（商品/库存/会员/订单） |
> | HTTP 层 | Express + SSE | 实现 Streamable HTTP 协议传输 |
> | 运维层 | cpolar 内网穿透 | 把本地 Server 暴露成公网 HTTPS 地址 |
>
> **亮点**：完整跑通 MCP SSE 协议（包括 body stream 兼容、API Key 鉴权、跨平台 Windows 二进制兼容）

## 三、现场演示话术（5 分钟走查）

### 演示第 1 句：查价格（30s）

打开浏览器 → 粘贴 URL → 进入对话 → 发送：

**"可口可乐 330ml 多少钱？"**

预期：智能体调用 `price_query` Tool，返回：可口可乐 330ml 单价 ¥2.50

**你说**：
> "可以看到，模型**自动选择**了 `price_query` 这个 Tool 去查价格——我没在提示词里硬编码调用哪个工具，这是 MCP 的核心：AI 自己看 Tool 描述决定调什么。"

### 演示第 2 句：查库存（30s）

**"美佳便利店还有几瓶可乐？"**

预期：调用 `inventory_query` 返回：美佳便利店可口可乐 330ml 库存 156 瓶

**你说**：
> "库存是动态数据，从 MCP Server 的 SQLite 实时查的，不是训练数据里的硬编码。"

### 演示第 3 句：查优惠（30s）

**"鲜多多门店今天有什么优惠？"**

预期：调用 `promo_list` 返回：满 30 减 5、第二件半价等

**你说**：
> "促销活动数据也是实时的，运营改了一次促销，AI 立刻就能查到——这就是 MCP 相比 LLM 预训练的核心优势。"

### 演示第 4 句：下单（1 分钟，重点）

**"帮我下个单：可口可乐×2、农夫山泉×1，送美佳门店自提。"**

预期：智能体分两步——
1. 调用 `create_order` 创建订单（含自动折扣）
2. 返回订单号、总价、自提码

**你说**：
> "这一步是关键：用户没说'我要下单'，但 AI **理解意图** 后自动选了 `create_order` 这个 Tool。
>
> 注意看返回结果：订单号、价格、优惠金额都是动态计算出来的。
>
> **顾客从来没打开 App，全在微信对话框里完成。**"

### 演示第 5 句：全栈视角（1 分钟）

切到管理后台 `http://localhost:3100/admin/`：

1. **登录** → 演示 admin / admin123
2. **MCP 模板页** → "看到没有，这 14 个 Tool 都是可视化勾选生成的"
3. **一键生成页** → "点这个按钮就能生成新的 MCP Server"
4. **实时监控页** → "调用量、延迟、错误率全在这里——这是给运营方看的"
5. **订单管理页** → "刚才演示创建的订单就在这里，来源标记为 `wechat_ai`，状态是'待支付'"
6. **支付状态** → "点击 process_payment Tool，生成微信支付链接，顾客点击就能支付"

**你说**：
> "系统分两半：
> - **左半** 顾客视角：微信里对话 → AI 自动调 MCP Tool → 生成支付链接 → 点击支付
> - **右半** 商家视角：可视化配置 + 实时监控 + 订单管理 + 支付状态跟踪
>
> 支付流程是关键：顾客在微信对话框里点击链接，直接跳转到微信支付页面，支付完成后订单状态自动更新。
>
> 这就是 MCP Forge 想做的事情：**让中小零售商像点外卖一样接入微信 AI，完成从下单到支付的完整闭环**。"

## 七、被问"项目难点"怎么答（技术向）

> **第一难：MCP 协议实战**
> - MCP 1.6.0 SDK 文档少，很多细节靠自己读源码
> - Express `express.json()` 全局中间件会消费 body stream，导致 MCP POST 失败——必须精确跳过 `/mcp` 路由
> - Streamable HTTP 协议的 SSE 长连接需要正确处理 connection / message 生命周期

> **第二难：跨平台兼容**
> - 本来是 macOS 写的，迁到 Windows：`better-sqlite3` 是原生模块需要 ABI 匹配
> - 解决方案：手动下载预编译 `.node` 二进制，确保 Node v22 ABI v127

> **第三难：公网穿透到 AI 平台**
> - 元器要求 HTTPS，本地只有 HTTP → 用 cpolar 内网穿透
> - 验证 5 句演示话术全部跑通，证明端到端链路真的可用
