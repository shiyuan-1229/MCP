# MCP Forge 项目建议方案

> 基于《2026 暑期 POC 课题同步：MCP Forge —— MCP 工厂》整理
> 更新日期：2026-07-02 | 新增科传业务背景及下周会议准备建议

---

## 一、核心判断

MCP 正在从"协议标准"演进为"AI Agent 经济的基础设施"。2026 年 6 月瑞幸、支付宝、微信密集发布 MCP/AI 支付能力，**以微信 Agent 为代表的新流量入口正在形成，企业不主动 MCP 化，就会被 AI Agent 跳过**。

**科传**作为智慧零售引领者，核心机会在于：
1. **帮助客户把遗留系统（ERP、CRM、POS、会员等）整合为 MCP 接口**，让这些系统能被 AI Agent 直接调用
2. **借助 微信 Agent 等 AI 入口**，为客户获取新流量，而不是被流量平台绕过
3. 在此基础上提供**知识库搭建、系统集成咨询等增值盈利服务**

MCP Forge 的机会不在于"做一个代码生成器"，而在于成为**企业把业务能力接入 AI Agent 的默认入口**。单打 MCP 工厂价值有限，必须和 Token 供应、Agent 中台、知识库打包成"企业 AI 能力套件"。

## 二、产品定位建议

### 2.1 一句话定位

> **让企业已有的 API、数据和服务，零代码/低代码地变成 AI Agent 可调用的标准 MCP Server。**

### 2.2 目标用户分层

| 用户层级 | 典型角色 | 核心诉求 | 付费意愿 |
|---------|---------|---------|---------|
| 种子层 | 开发者、独立 ISV | 快速把自研 API 封装成 MCP，验证想法 | 低，适合 Freemium |
| 增长层 | SaaS 厂商、系统集成商 | 让自己的产品被 Claude/Cursor/WorkBuddy 等 Agent 调用 | 中，按量/按功能订阅 |
| **核心层** | **商业地产、零售、物业等甲方 IT** | **把内部系统（会员、订单、设备、工单）开放给 AI Agent** | **高，打包年费方案** |

### 2.4 科传的独特切入点

作为智慧零售引领者，科传已有深厚的零售行业客户基础。MCP Forge 应围绕科传的核心能力展开：

1. **遗留系统整合**：零售客户的旧 ERP/CRM/POS 系统往往 API 不规范甚至没有 API。需要提供"适配器"模式——先接入、再封装、再 MCP 化。
2. **知识库增值服务**：客户需要的不只是 MCP，还有背后支撑 AI Agent 的知识库（商品知识、政策文件、FAQ 等）。这是独立的盈利服务项。
3. **微信 Agent 流量红利**：微信推出 AI 专属卡/Agent 能力后，零售客户最紧迫的需求就是"我的业务怎么接入微信 AI"——这正是 MCP Forge 的销售抓手。

### 2.3 价值主张

1. **降门槛**：不需要理解 MCP 协议细节，上传 Swagger 即可生成 Server。
2. **可验证**：生成的 Server 能直接在 Claude Desktop / Cursor / WorkBuddy 中跑通。
3. **可托管**：不只是生成代码，还提供云端运行、监控、鉴权、版本管理。
4. **可闭环**：和 Token、Agent、知识库打包，形成"企业 AI 能力中台"。

## 三、技术实现方案

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     MCP Forge 产品层                          │
│  [Swagger 导入]  [手动配置]  [DB Schema 导入]  [示例模板]        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     协议转换引擎                              │
│  OpenAPI/Swagger 解析 → 参数类型映射 → MCP Tool Schema 生成    │
│  Auth 鉴权适配（API Key / OAuth2 / JWT）→ 错误码标准化         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     生成与运行层                              │
│  多语言代码生成（Node.js / Python / Java）                    │
│  沙箱运行环境 + 云端托管 + 日志/监控/告警                      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     验证与分发层                              │
│  Claude Desktop 配置导出 / Cursor 集成 / WorkBuddy 连接器     │
│  MCP Server 市场 / 版本管理 / 使用统计                         │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 关键技术点

| 模块 | 说明 | 优先级 |
|-----|------|-------|
| Swagger/OpenAPI 解析器 | 支持 2.0/3.0/3.1，处理 path、method、parameters、requestBody、responses | P0 |
| Schema 映射引擎 | 把 JSON Schema 转换为 MCP Tool 的 inputSchema，处理嵌套对象、枚举、文件上传 | P0 |
| 多运行时支持 | 首期建议聚焦 Node.js（与 Claude Desktop 生态最契合），后续扩展 Python/Java | P1 |
| 认证鉴权中间件 | 统一处理 API Key、OAuth2、JWT，企业客户可配置权限边界 | P0 |
| 沙箱执行 | 防止用户生成的 Server 代码存在安全风险，建议用容器或 WebAssembly 沙箱 | P1 |
| 调试台 | 可视化测试每个 Tool 的输入输出，模拟 MCP Client 调用 | P0 |
| 版本同步 | 当源 API 变更时，自动检测并提示更新 MCP Server schema | P1 |

### 3.3 参考瑞幸 MCP 的对标分析

建议 POC 阶段专门研究瑞幸 MCP 的实现：
- 暴露了哪些业务能力？（点单、会员、优惠券、门店查询？）
- Tool 的粒度如何设计？（是按业务模块拆分，还是按 API 平铺？）
- 认证方式是什么？
- 在 Claude Desktop 中的实际体验如何？

这些结论直接决定 Forge 的默认生成策略。

## 四、POC / MVP 范围建议

暑期 POC 建议聚焦"**最小可验证工厂**"，不要贪大：

### 4.1 MVP 功能清单

1. **导入**：支持上传 Swagger JSON/YAML，自动解析 API 列表。
2. **选择**：用户勾选要暴露为 MCP Tool 的接口，可编辑名称、描述、参数。
3. **生成**：一键生成 Node.js 版 MCP Server 代码。
4. **运行**：提供云端托管沙箱，生成可访问的 SSE/stdio URL。
5. **验证**：导出 Claude Desktop / Cursor 配置，支持一键复制粘贴验证。
6. **示例**：内置 1-2 个示例 API（如天气查询、订单查询），让新用户 5 分钟跑通。

### 4.2 非 MVP（后续迭代）

- 多语言代码生成
- 私有化部署
- MCP Server 市场
- API 变更自动同步
- 复杂认证（OAuth2 流程）
- 数据库直连生成 MCP Resources

### 4.3 成功标准

- 一个真实客户的 Swagger 能在 30 分钟内完成 MCP 化
- 生成的 Server 能在 Claude Desktop 中成功调用
- 至少完成 3 个不同行业的 demo（零售、物业、商业地产）

## 五、商业模式与定价建议

### 5.1 两种收费模式

根据你提到的方向，以下两种模式可以并行推进：

| 模式 | 说明 | 适合阶段 | 优点 | 缺点 |
|------|------|---------|------|------|
| **按注册计费** | 客户按接入的业务系统数量/API 数量收费，月付或年付 | 初期获客 | 收入可预期，客户成本可控 | 无法体现调用量差异 |
| **按调用计费** | 客户按 AI Agent 调用 MCP 接口的次数收费，按量计费 | 跑通后 | 弹性计费，客户按效果付费 | 初期收入不稳定 |

**建议的组合策略**：
- **基础费（注册制）**：覆盖平台使用和系统集成成本
- **浮动费（调用制）**：按实际调用量加收，与客户业务增长正相关
- 初期获客可以"注册费打折，调用费后付"降低门槛

### 5.2 三阶段路线

```
阶段一：开发者工具（0-6 个月）
  └── Freemium：每月免费 3 个 API 转换
  └── Pro：$29/月，无限转换 + 私有部署 + 调试台

阶段二：企业 AI 能力套件（6-12 个月）
  └── TokenRelay + Agent 中台 + 知识库 + MCP Forge 打包
  └── 按企业年费，10-50 万/年起步

阶段三：MCP 服务市场（12 个月以上）
  └── 第三方发布 MCP Server，平台抽佣 10-20%
  └── 类似早期 App Store / 微信小程序生态
```

### 5.2 定价策略要点

- **先以套件卖大单，再以工具铺生态**：企业客户买单的是"AI 能力整体交付"，不是单个代码生成器。
- **开发者版用于建立标准**：让更多 ISV 免费/低价使用，形成"MCP 化"的行业默认路径。
- **抽佣模式需要流量基础**：不要过早做市场，等 WorkBuddy 或其他平台的 Agent 调用量起来后再启动。

## 六、与现有能力的协同

你们已经落地了**微信 AI 专属卡 / AI 支付**能力（见配图），这是极大的差异化优势：

| 现有能力 | 与 MCP Forge 的结合点 |
|---------|---------------------|
| 微信 AI 支付 | 商家的订单、库存、会员系统通过 MCP Forge 暴露后，AI Agent 可直接"查库存 → 下单 → 微信支付"全链路闭环 |
| TokenRelay | MCP Server 调用外部 API 时统一走 Token 供应，解决企业调用成本分摊问题 |
| Agent 中台 | 让企业快速搭建专属 Agent，MCP Forge 负责把企业业务能力喂给 Agent |
| 知识库 | 企业文档、FAQ 通过 MCP Resources 暴露，Agent 可实时检索 |

**建议打造一个标杆场景**：AI 助手帮用户找附近烧烤店 → 调用商家 MCP（菜单/库存/排队）→ AI 推荐套餐 → 用户确认 → 微信支付完成。这个 demo 极具说服力。

## 七、关键风险与应对

| 风险 | 影响 | 应对策略 |
|-----|------|---------|
| MCP 协议快速迭代 | 生成的 Server 可能不兼容新版 Client | 抽象一层"协议适配层"，屏蔽底层版本差异 |
| 安全问题 | 企业 API 暴露给 AI 可能被越权调用 | 最小权限原则、调用审计、沙箱隔离、人工确认敏感操作 |
| 巨头自研 | Claude/Cursor 自己做 MCP 生成工具 | 专注中文企业市场、打包本地化能力（微信/支付宝/企业微信） |
| 客户认知不足 | 不知道为什么要 MCP 化 | 用标杆 demo 和客户共创，量化"接入 AI Agent 后订单转化率提升" |
| 生成质量不稳定 | API 太复杂时转换失败 | 先做常见 REST API 模式，逐步覆盖 edge case；提供人工修正入口 |

## 八、下周会议准备建议

基于你提到的下周会议将讨论**产品方向**与**盈利模式**，建议会议前重点准备以下内容：

### 8.1 议题一：产品方向 —— 做什么、不做什么

**完整方向一览（共 5 个）：**

| 方向 | 核心定位 | POC 周期 | 壁垒 | 可复制性 | 客单价 | 适合谁 |
|------|---------|:--------:|:----:|:-------:|:------:|-------|
| **A·纯 MCP 工厂** | API→MCP 一键转换工具，轻量 SaaS | 4-6 周 | ⭐低 | ⭐⭐⭐高 | ¥1-3万 | 想快速上车但无所谓壁垒的团队 |
| **B·工厂 + 知识库** | A + 知识库搭建/管理增值服务 | 8-12 周 | ⭐⭐中 | ⭐⭐中 | ¥5-15万 | 想比纯工具多赚服务费 |
| **C·企业 AI 能力套件** | B + 旧系统集成 + Agent 中台 + Token 供应打包 | 12-16 周 | ⭐⭐⭐高 | ⭐中低 | ¥10-50万 | 做长期大客户、建生态壁垒 |
| **D·行业模板库 ☆** | 预装零售行业业务模板，客户拿来即用 | 6-10 周 | ⭐⭐中 | ⭐⭐⭐极高 | ¥2-6万/年 | 快速铺客户，中小型零售连锁 |
| **E·安全网关平台 ☆** | MCP Server 生产运维平台（安全/审计/监控/计费） | 10-14 周 | ⭐⭐⭐极高 | ⭐⭐中 | ¥10-30万/年 | 大型集团，安全合规诉求强 |

**推荐组合策略**：

```
短期获客 ──→  D·行业模板库（门槛低、签单快）
长期壁垒 ──→  E·安全网关平台（技术壁垒高、续费稳）
终极愿景 ──→  C·企业 AI 能力套件（生态化、绑定深）
```

> D 打头阵快速获客 → 客户成长后升级 E 做安全管控 → 最终演进到 C 全栈套件

每个方向的具体应用场景详见《MCP_Forge_产品方向应用场景.md》。

**建议会议通晒材料清单**：
- 瑞幸 MCP 竞品分析报告（1 页摘要）
- 2-3 个零售客户的初始需求调研（至少电话沟通）
- MCP Forge 最小原型线框图（工具 + 知识库模块）
- 技术可行性验证（Core MCP 协议是否已跑通）

### 8.2 议题二：盈利模式 —— 怎么定价

| 定价模式 | 目标客户 | 预期客单价 | 注重点 |
|---------|---------|-----------|-------|
| 按注册/年费 | 中小型零售商 | ¥5k-¥20k/年 | 降低决策门槛 |
| 按调用量计费 | 中大型客户 + 高并发场景 | ¥0.1-¥1/千次调用 | 体现按效果付费 |
| 打包模式（注册+调用） | 全部客户 | ¥20k-¥100k/年起 | 最推荐，兼顾稳定性和弹性 |
| 纯定制集成服务 | 大型集团 | ¥100k-¥500k/项目 | 高利润但不可规模化 |

### 8.3 会议前需要确认的关键问题

1. **目标客户优先级**：三个月内先打哪类客户？（现有科传客户？新零售品牌？还是物业商业地产？）
2. **对手分析**：市面上已有的 MCP 生成工具（如 Anthropic 官方工具、开源项目）处于什么阶段？我们做什么差异化？
3. **资源投入**：POC 阶段能投入多少人？建议至少 1 产品 + 1 后端 + 1 前端。
4. **成功标准**：POC 结束后，什么算"成功"？（签下一个客户？跑通技术验证？拿到融资？）

## 九、下一步行动建议

1. **本周**：成立 2-3 人 POC 小组，明确一名产品经理 + 一名后端 + 一名前端。

5. **第四周**：准备 POC 汇报材料，包含 demo 视频、技术架构、商业模式推演。

## 十、总结

MCP Forge 的**最大价值不是"代码生成"，而是"让企业业务能力被 AI Agent 调用"**。在 2026 年 MCP 元年 + 微信 Agent 新流量入口的双重窗口期，**科传的核心战略**应该是：**用 MCP 工厂做抓手，帮客户整旧系统 + 建知识库，以注册和调用双计费模式变现，抢占微信 AI 生态的先发红利**。

当前最重要的不是功能多全，而是**尽快跑通一个真实客户的端到端闭环**，让这个客户愿意为你背书。
